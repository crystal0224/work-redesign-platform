# Work Redesign 워크샵 파일럿 - Synthetic Users 설계 프로세스

## 0. Synthetic Users란?

### 0.1 정의

**Synthetic User** (합성 사용자)는 실제 사용자의 특성, 행동 패턴, 의사결정 프로세스를 데이터 기반으로 정교하게 모델링한 AI 에이전트입니다.

### 0.2 학술적/산업적 배경

**활용 분야**:
- **UX Research**: Google, Microsoft 등 빅테크의 대규모 UI 테스트
- **HCI (Human-Computer Interaction)**: 사용자 연구 비용 절감
- **Medical Training**: 환자 시뮬레이터 (증상, 반응 패턴 재현)
- **Customer Service**: 고객 페르소나 기반 챗봇 테스트

**핵심 원리**:
1. **데이터 기반 모델링**: 실제 사용자 데이터 → 페르소나 속성 (300+ 데이터 포인트)
2. **AI 에이전트 구현**: LLM을 활용한 자연스러운 행동/응답 생성
3. **시나리오 기반 시뮬레이션**: 실제 환경에서 과제 수행 + 피드백
4. **통계적 유의성**: 충분한 샘플 크기 (n=30) + 세그먼트 다양성

### 0.3 왜 Synthetic Users를 사용하는가?

| 한계점 | 실제 사용자 파일럿 | Synthetic Users 시뮬레이션 |
|--------|-------------------|---------------------------|
| **모집의 어려움** | 특정 세그먼트 (예: Beginner 제조팀장) 찾기 어려움 | 세그먼트별 정확히 30명 구성 가능 |
| **비용** | 1회 $3,000+ | 무제한 반복 $1.50/회 |
| **일관성** | 컨디션, 시간대에 따라 반응 변동 | 동일 조건 재현 가능 |
| **반복성** | UI 수정 후 재테스트 비용/시간 부담 | UI 변경 시마다 즉시 재실행 |
| **세밀도** | 정성 피드백 수집 제한적 | 단계별, 요소별 상세 피드백 |

**신뢰성 확보 조건**:
- ✅ 현실적 페르소나 (실제 직무, 도구, pain points 기반)
- ✅ Enhanced Prompting (System identity + CoT + Few-shot)
- ✅ 검증된 방법론 (실제 HRD 파일럿 프로세스 재현)
- ✅ Generic 답변 방지 (부서별, 성숙도별 차별화)

## 1. Synthetic Users 설계 전략 (Persona Extraction)

### 1.1 표본 설계 원칙
- **대상**: SK그룹 계열사 팀장 (팀원 아닌 리더 개인 교육)
- **표본 크기**: 30명 (통계적 유의성 + 실행 가능성)
- **표본 구성**: SK그룹 실제 직무 분포 반영

### 1.2 직무별 할당 근거
| 직무 | 인원 | 근거 |
|------|------|------|
| Marketing & Sales | 5명 (17%) | SK플래닛, SK텔레콤 등 고객 대면 직군 |
| Operations | 8명 (27%) | SK E&S, SK에코플랜트 등 생산/운영 직군 (최다) |
| R&D | 5명 (17%) | SK하이닉스, SK이노베이션 등 기술개발 직군 |
| Staff | 12명 (40%) | 모든 계열사 공통 지원 직군 (HR 3, Finance 2, IT 4, Strategy 3) |

**판단 근거**: SK그룹은 제조/에너지/인프라 중심이므로 Operations 비중 최대. Staff는 모든 계열사에 필수이므로 세분화하여 다수 배정.

## 2. Synthetic Users 정의 프로세스 (Persona Definition)

### 2.1 Phase 1: 기본 프레임워크 설계
```typescript
interface Persona {
  // 신원: id, 이름, 소속, 직급
  // 팀 구성: 팀 규모, 디지털 성숙도
  // 업무: 주요 과제, 사용 도구
  // 행동 예측: 워크샵 태도, 중단 위험도
}
```

### 2.2 Phase 2: 팀장 관점 재정의 (1차 피드백 반영)
**변경 사유**: "팀원이 다 참여하는 게 아니라 팀장만 워크샵 참여"
**Synthetic User 일관성**: 팀장 본인의 워크샵 경험 vs 팀원들에 대한 고려 명확히 구분

**주요 변경**:
- `team.members[]` 삭제 → `team.composition` (팀 구성 설명)으로 변경
- `painPoints` → 팀장이 팀 운영하며 느끼는 어려움으로 재정의
- `concerns` → 워크샵 참여 우려 (팀에 적용 가능성, 시간 부족 등)
- `leaderProfile` 추가: 팀장 경력, 이전 역할, 리더십 스타일

**예시 변경**:
```
[Before] concerns: ["새로운 툴 학습이 부담"]
[After] concerns: ["워크샵에서 배운 내용을 팀에 어떻게 적용할지 막막함"]
```

### 2.3 Phase 3: 업무 구조화 속성 추가 (2차 피드백 반영)
**추가 사유**: "업무의 구조화 정도도 정리"

```typescript
workStructure: {
  level: '비구조화' | '반구조화' | '고도구조화',
  description: string // 역할 분장, 프로세스 문서화 상태
}
```

**분류 기준**:
- **비구조화**: 역할 유동적, 프로세스 문서 없음 (예: 스타트업형 마케팅팀)
- **반구조화**: 역할은 있으나 세부 프로세스는 재량 (예: 캠페인팀)
- **고도구조화**: 역할/프로세스 명확히 정의됨 (예: 제조/품질관리)

### 2.4 Phase 4: 직무 균형 조정 (3차 피드백 반영)
**문제**: Staff 15명으로 쏠림
**해결**: Operations 5→8명 증원, Staff를 4개 하위 직무로 세분화

## 3. 핵심 속성 정의 기준

### 3.1 디지털 성숙도 (팀 단위)
- **Beginner**: Excel 수준, 협업툴 미사용
- **Intermediate**: 클라우드 도구 사용, 자동화 초기 단계
- **Advanced**: AI 도구 일부 사용, 데이터 기반 의사결정
- **Expert**: AI 네이티브, 자동화 고도화

### 3.2 중단 위험도 (dropoutRisk %)
- **10-20%**: 기술 친화적, 높은 인내심
- **30-40%**: 보통 수준
- **50-70%**: 변화 저항 높음, 시간 압박 심함

**산정 공식**:
```
dropoutRisk = (changeResistance) × (1 - patience/10) × (1 - techSavvy/10) × 100
```

### 3.3 문제 예상 단계 (problemSteps)
**연계 로직**: 페르소나 특성 → UI 스펙의 expected_issues 매핑
- Beginner → Step 1,2 (용어 이해)
- Large Team → Step 5 (업무량 입력 복잡)
- 비구조화 → Step 4 (역할/과제 정의 어려움)

## 4. 검증 및 현실성 확보

### 4.1 현실성 체크리스트
- ✅ 실제 SK 계열사명 사용 (SK플래닛, SK하이닉스 등)
- ✅ 구체적 업무량 (예: "주당 8시간 수작업", "월 5-8개 캠페인")
- ✅ 실제 툴명 (Excel, SAP, Slack, Figma 등)
- ✅ 직군별 전형적 pain point (마케팅: 데이터 통합, 제조: 실시간 현황 파악)

### 4.2 다양성 확보
- 팀 규모: 5명~50명 (소규모~대규모)
- 팀장 경력: 1년~15년
- 디지털 성숙도: 4단계 고르게 분포
- 리더십 스타일: 수평적, 수직적, 데이터 기반, 관계 중심 등

## 5. Synthetic Users 구현 방법

### 5.1 AI 페르소나 시뮬레이션 (Enhanced Prompting)
**System Identity (Enhanced Prompting v3)**:
```typescript
`당신은 ${persona.name} 팀장입니다.

# 당신의 정체성
- 회사: ${persona.company}
- 부서: ${persona.department} (${persona.category})
- 팀장 경력: ${persona.leaderProfile.yearsInRole}년

## 팀 상황
- 팀 규모: ${persona.team.size}명
- 성숙도 분포: ${persona.team.maturityDistribution}

## 매일 하는 업무
${persona.work.mainTasks.map(t => `- ${t}`).join('\n')}

## Pain Points (매일 겪는 어려움)
${persona.work.painPoints.map(p => `- ${p}`).join('\n')}

## 업무 구조화
- ${persona.work.workStructure.level}: ${persona.work.workStructure.description}
`
```

**+ Chain of Thought**: 답변 전 페르소나 입장에서 생각
**+ Few-shot Examples**: 부서별 좋은/나쁜 답변 예시 제공
**+ Temperature Tuning**: digitalMaturity + changeResistance 기반 (0.6-0.9)

### 5.2 결과 분석 세그먼트
- 디지털 성숙도별 완료율 비교
- 직무별 문제 단계 분석
- 업무 구조화 수준별 소요 시간 분석

## 6. 결과물: 30명의 Synthetic Users

**최종 파일**: `2-personas/personas-v3.ts`

### 6.1 규모 및 구성
- **30명 Synthetic Users**: 각 페르소나당 300-400줄 상세 정의
- **8개 직무 카테고리**: Marketing, Sales, Operations, R&D, HR, Finance, IT, Strategy
- **15개 핵심 속성**: 신원, 팀장 프로필, 팀 구성, 업무 특성, 예상 행동, 성격

### 6.2 데이터 포인트
각 Synthetic User당 **300+ 데이터 포인트**:
- 리더십 스타일, 팀장 경력, 이전 역할
- 팀 규모, 구성, 디지털 성숙도 분포
- 주요 업무 5가지, 사용 도구 6개, Pain points 3-4개
- 자동화 니즈, 업무 구조화 수준, 변화 저항, 인내심 등

### 6.3 검증된 현실성
- ✅ **실제 SK그룹 기반**: 계열사명, 직무, 도구 (SAP, Salesforce, Figma 등)
- ✅ **구체적 수치**: "주당 8시간 수작업", "월 5-8개 캠페인" 등
- ✅ **다양성 확보**: 팀 규모 5-50명, 팀장 경력 1-15년, 성숙도 4단계
- ✅ **워크샵 시나리오**: 팀장만 참여, 팀 특성 반영, 미래 적용 우려

### 6.4 AI 시뮬레이션 실행 가능성
- Claude Sonnet 4.5 (사전/사후 인터뷰)
- Claude Haiku 4.5 (워크샵 실행, 중간 체크인)
- Enhanced Prompting으로 generic 답변 방지
- 부서별 차별화된 피드백 생성
